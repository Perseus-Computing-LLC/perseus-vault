# Perseus Vault durable cognition strategy

Date: July 20, 2026
Owner: Thomas Connally

## Executive summary
Atlassian appears to be building a managed enterprise memory plane optimized for product-native relevance, work-graph context, and permission-aware retrieval. Perseus Vault should intentionally differentiate rather than compete.

Vault should be the durable memory substrate for long-horizon continuity, explicit user control, provenance, correction, supersession, and explainable served-memory support.

## What Vault should optimize for
- explicit memory taxonomy
- default precedence rules
- source anchors to external systems
- correction and supersession workflows
- long-horizon retention and decay controls
- served-memory APIs for synthesis and explainable recall

## Non-goals
- enterprise permission inheritance parity
- graph-store parity with Teamwork Graph/Flock
- tenant-wide managed shared-memory infrastructure
- product-native memory editor parity

## Priority roadmap
### Now
- formalize memory taxonomy
- define default precedence
- improve source anchoring

### Next
- correction and supersession improvements
- long-horizon retention and decay controls
- served-memory APIs for synthesis flows

### Later
- policy packs for memory behavior
- richer reinforcement and consolidation controls
- external import paths without schema lock-in

## Success metrics
- canonical memory classes documented
- precedence rules referenced in recall design
- source anchor model documented with examples
- at least one served-memory API validated against a synthesis flow

## Bottom line
Vault should be the durable, auditable, user-steerable memory substrate that complements product-native enterprise memory.
